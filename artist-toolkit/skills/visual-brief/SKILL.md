# Visual Brief — Méthode

## Rôle de la skill

Cette skill **codifie le look** d'un artiste : elle transforme une identité visuelle implicite
(dispersée dans un EPK, des covers, un feed) en un **document de référence structuré**, lisible
par un humain comme par les autres skills. Elle couvre **trois tâches précises**, et rien d'autre :

1. Construire le document de référence d'identité visuelle.
2. Le maintenir et le faire évoluer.
3. En dériver un brief pour un consommateur précis (Brand Kit Canva, skill de génération visuelle,
   ou livrable graphique ponctuel).

Elle **codifie, elle ne fabrique pas**. Elle ne génère aucune image ni vidéo, n'écrit aucun copy.
Son livrable est toujours du **texte structuré** : le doc d'identité, ou un brief qui en découle.

## Principe — méthode, pas identité

La skill est agnostique de l'artiste, pour servir n'importe quel projet musical. Elle ne code
**jamais** en dur une couleur, une police, un logo, un thème ou une plateforme. Avant toute
analyse :

- Lire le **positionnement** et la **voix** du projet dans ses instructions : l'identité visuelle
  doit les servir, pas les contredire (p. ex. un positionnement « mystère / brut » interdit une
  charte léchée et sur-produite).
- Rassembler les **assets visuels** disponibles dans le projet : EPK, pochettes, visuels joints,
  captures du feed Instagram/TikTok, anciens posts. Si rien n'est exploitable, le demander avant
  d'inventer quoi que ce soit.
- Ne jamais combler un trou par une « bonne pratique » de branding générique. Ce qu'on observe
  réellement dans les assets fait foi ; ce qui manque est signalé comme **à décider**, pas deviné.

## Posture

- **Honnête avant d'être gentil.** Le but est une charte utilisable, pas un compliment. Si
  l'identité actuelle est incohérente (deux logos, trois polices, aucune règle), le dire et
  proposer un arbitrage. Pas de remplissage.
- **Distinguer l'observé du décidé.** Tout élément du doc est soit *constaté* dans un asset (avec
  la source), soit *proposé* à l'artiste pour validation. Ne jamais présenter une proposition
  comme un fait établi.
- **Socle stable vs sortie.** L'identité visuelle appartient à l'univers permanent du projet, pas
  à une sortie. Une cover de single peut moduler la charte, jamais la remplacer. Le doc sépare
  explicitement le socle (stable) des déclinaisons par sortie.

## Fonction 1 — Construire le doc d'identité visuelle

**Déclencheur** : « définis / extrais mon identité visuelle », « code ma charte », « formalise
mon look ».

**Inputs nécessaires** :
- Les assets visuels du projet (voir Principe). À défaut, demander à l'artiste de fournir : 2-3
  covers, des captures du feed, le logo, et toute photo officielle.
- Le positionnement et la voix, lus dans les instructions du projet.

**Action** : examiner les assets, puis remplir les sections canoniques ci-dessous. Pour chaque
élément, noter la **source** (« observé sur la cover X », « EPK p.2 ») ou le marquer **à décider**.

Sections canoniques du doc :

1. **Palette** — chaque couleur : nom, code hex (relevé ou approché depuis l'asset, signalé comme
   approché le cas échéant), et **rôle** (fond / accent / texte). Pas une liste de couleurs : une
   hiérarchie d'usage.
2. **Typographie** — familles utilisées, casse (p. ex. capitales condensées), usage par niveau
   (titres / sous-titres / corps), et les associations autorisées.
3. **Logos & marques** — logo principal, monogramme, pictogrammes récurrents ; règles d'usage :
   versions couleur, taille mini, zone de protection, ce qu'on ne fait pas avec.
4. **Traitement photo / vidéo** — noir & blanc vs couleur, grain, contraste, lumière, cadrages
   signature. Si un marqueur d'identité du projet est visuel (un instrument, un objet, un lieu),
   préciser comment il est filmé/cadré.
5. **Composition & mise en page** — grille, marges, alignements, principe de fond (uni / papier /
   texturé), gestion de l'espace négatif.
6. **Motifs & iconographie** — éléments graphiques récurrents (formes, pictos, curseurs, textures)
   et leur sens.
7. **Ton visuel** — en une ligne, l'effet recherché, rattaché au positionnement du projet (p. ex.
   « brut, mystérieux, jamais publicitaire »). C'est le filtre qui tranche les cas limites.
8. **Do / Don't** — garde-fous concrets. Inclure systématiquement les pièges déduits du
   positionnement (p. ex. « ne pas appliquer la charte graphique au contenu vidéo signature qui
   doit rester brut »).
9. **Déclinaisons par sortie** — comment le socle se module d'une sortie à l'autre (variation de
   couleur d'accent, motif propre à un album) **sans** casser les sections 1-7.

**Output** : le document complet, prêt à être enregistré dans le projet comme **document de
référence** (pas comme skill). Terminer par une courte liste des points **à décider** restants,
pour que l'artiste les tranche avant que la charte soit figée.

## Fonction 2 — Maintenir et faire évoluer le doc

**Déclencheur** : « mets à jour ma charte », « j'ai de nouveaux visuels », « le look a bougé pour
le prochain album ».

**Inputs nécessaires** : le doc d'identité existant + le ou les nouveaux assets / changements.

**Action** :
- Comparer le nouvel élément au socle. Trois cas : *cohérent* (rien à changer), *déclinaison*
  (va en section 9, le socle ne bouge pas), *dérive* (entre en conflit avec le socle → le signaler
  et demander un arbitrage : on adapte l'asset, ou on fait évoluer le socle en conscience).
- Ne jamais réécrire le socle en silence. Tout changement de section 1-7 est explicite et validé.

**Output** : la version mise à jour, avec un court **changelog** (ce qui a changé, où, pourquoi).

## Fonction 3 — Dériver un brief pour un consommateur

**Déclencheur** : « prépare mon Brand Kit », « brief visuel pour [cover / story / clip] », « passe
ma charte à [skill visuelle] ».

**Inputs nécessaires** : le doc d'identité + le consommateur visé + le livrable concerné.

**Action** : traduire le doc dans le format attendu par la cible. Ne **rien inventer** au-delà du
doc ; si le livrable exige une décision absente du doc, la remonter (et la verser au doc via la
Fonction 2 si elle est tranchée). Cibles typiques :

- **Outil de design** (Brand Kit, palette limitée, injection dans un prompt de génération, ou
  outil de design en chat) — sortir un **bloc d'identité copiable** : hex avec leur rôle, familles
  typographiques + casse, gestion des logos. Si l'outil a un vrai Brand Kit (souvent payant), y
  verser la spéc. Sinon : ce même bloc s'**injecte dans le prompt** de génération, et la consistance
  durable passe par des **templates réutilisables** (un par format) qui portent déjà couleurs,
  polices et logo. Garde-fou palette : noir et blanc sont des défauts universels — dans une palette
  à places limitées, ne pas gâcher un emplacement pour eux, le réserver aux couleurs de marque non
  standard. La skill produit le bloc/la spéc ; la création du Brand Kit ou des templates reste une
  action de l'artiste.
- **Skill de génération visuelle** (prompts image/vidéo) — sortir un bloc d'identité réutilisable
  (palette, traitement photo, cadrages signature, motifs, do/don't) que la skill visuelle injecte
  dans ses prompts. La skill `visual-brief` **n'écrit pas le prompt final** : elle fournit la
  matière, la skill visuelle compose.
- **Livrable graphique ponctuel** — un cahier des charges court pour une cover / story / template
  précis : contraintes de la charte + ce qui est libre. La fabrication du visuel ne lui appartient
  pas.

**Output** : le brief au format de la cible, strictement dérivé du doc, avec les éventuels points
manquants signalés.

## Frontières avec les autres skills

`visual-brief` **codifie et oriente**. Dès qu'il faut **fabriquer**, elle passe la main :

- Générer des images (character sheets, plates, visuels) → la skill de génération d'images du projet.
- Générer des prompts vidéo cinématiques → la skill de génération vidéo du projet.
- Évaluer / critiquer une proposition visuelle → skill **creative-coach**.
- Écrire les légendes, accroches, CTA → skill de **contenu social**.

Le lien avec les skills visuelles est une **dépendance** (le brief les alimente), pas une fusion :
`visual-brief` est une skill de carrière (elle lit le projet, codifie l'identité), les skills
visuelles sont des moteurs de génération généralistes. Ne pas empiéter sur leur rôle.

## Ressources & écriture

- Le doc d'identité est un **document de référence** stocké dans le projet, distinct des fichiers
  de skill. La skill le **lit** et propose son **contenu** ; elle ne s'auto-modifie pas.
- Lecture seule sur les assets et les trackers du projet.
